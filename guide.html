// COMMUNITY GUIDE

How r/factorymustgrow operates — moderation, standards, and how we interact.

Engineers building software as if it were Factorio. We share our production alpha like we are Taiichi Ohno. The factory must grow.

This community is ~80% software factory content — CI/CD pipelines, automated code review, deployment systems, infrastructure as code, throughput reports, post-mortems — and ~20% Factorio-inspired memes that hit different when you've also debugged a deploy pipeline at 2am.

// THE RULES

RULE 01
Production reports, not press releases
Share real production data, post-mortems, and throughput numbers. Not marketing copy, not LinkedIn posts, not corporate announcements. If it reads like a press release, it doesn't pass QC.
RULE 02
Link to your blueprint too
Share the configs, the repo, the architecture diagram. "Look what I built" is fine. "Here's how I built it and here's the repo" is what makes this sub compound in value. The blueprint is the alpha.
RULE 03
Spaghetti factories are factories
No elitism. Everyone starts somewhere. A bash script that deploys your side project is a factory. A messy Factorio base that produces science is a factory. Gatekeeping gets flagged by QC.
RULE 04
Open a PR to change moderation
Our moderation standards are source-controlled on GitHub. Want to change a rule? Want to adjust a QC check? Open a pull request. This is a factory — we version-control everything, including governance.
RULE 05
The factory must grow
Share what you learned, not just what you built. Each shared blueprint makes the next builder faster. The sub's value compounds when people open-source their factory designs.

// HOW MODERATION WORKS

This subreddit runs itself. Moderation is automated quality control, not human gatekeeping.

We use QC checks — markdown files in our GitHub repo that define what "good" looks like. An AI reads each post against these checks and flags quality control issues. Humans review the flagged items. Everything else flows through.

Think of it like a Factorio production line: items (posts) move along the belt. QC inserters pull out items that don't meet spec. Everything else ships. The factory doesn't stop to inspect every item — it catches the defects.

What happens when a post gets flagged

QC doesn't punish — it helps you ship. When a post gets flagged, you'll get a friendly QC comment explaining which check it didn't pass and why. Then it'll give you this:

Paste this into your favorite coding agent

Copied!Help me improve my post based on the r/FactoryMustGrow quality
control standards. Clone https://github.com/continuedev/factorymustgrow
and run the checks in .continue/checks/ against my draft.
Guide me through fixing anything that doesn't pass, and explain
why each standard matters so I learn to write in line with the
community voice.

Works with Claude Code, Cursor, Continue, Copilot, Windsurf — any agent that can clone a repo and read markdown.

Your agent will pull our QC checks, run them against your draft, and walk you through adjusting anything that doesn't pass. It's not about gatekeeping — it's about helping you write in the voice of the community. The checks explain why each standard exists, so you learn the factory's conventions, not just comply with rules.

The flow:

  1. You get a QC flag with a kind explanation of what didn't pass.
  2. You paste the prompt into your coding agent with your draft.
  3. The agent clones the repo, reads the checks, and guides you through revisions — explaining why each standard matters.
  4. You update your post. It passes QC. The factory grows.

If the flag is wrong, you can dispute it — and if the check itself is bad, you can PR a fix to the check. The QC system improves the same way the factory does: through iteration.

We want all people and agents to be part of this community. Whether you write your posts by hand or your AI drafts them, the same QC standards apply. The checks don't care who wrote it — they care that it ships production alpha.

Changing the rules

The moderation standards live at github.com/continuedev/factorymustgrow. Everything is in .continue/checks/.

To change a rule or add a check:

# Fork the repo
git clone https://github.com/continuedev/factorymustgrow
cd factorymustgrow

# Edit or add a check
vim .continue/checks/your-new-check.md

# Open a PR
gh pr create --title "Add check for [whatever]"

The community reviews the PR. If it passes, the check goes live. This is how a factory evolves its own QC standards.

// HOW WE INTERACT

Posting standards

  • Use the right flair. Blueprint for guides, Throughput for metrics, Bottleneck for help, Spaghetti for experiments, Alert for post-mortems. Flair is how the factory sorts items on the belt.
  • Be specific. "I made my pipeline faster" is noise. "I cut deploy time from 12 min to 90 sec by parallelizing test stages — here's the config" is production alpha.
  • Link to your blueprint. If you built something, share the repo, the config, the architecture diagram. The blueprint is more valuable than the screenshot.
  • Failures are welcome. Post-mortems are some of the highest-value content. "Here's what I tried that didn't work and why" saves the next builder time.

Commenting standards

  • Add to the blueprint. If someone shares a config and you've solved the same problem differently, share your approach. The best threads are production alpha stacking on production alpha.
  • Ask specific questions. "Why did you choose X over Y?" is better than "nice post." Specific questions invite specific answers.
  • Be direct, not harsh. "This won't scale because X — here's what I'd try instead" is good. "This is wrong" with no alternative is not.
  • Spaghetti is valid. Don't trash someone's working system because it's not elegant. If it produces output, it's a factory.

What doesn't belong

  • Press releases. If you're a company announcing a product, this isn't the venue. If you're an engineer at that company sharing how you built it — with configs and real numbers — welcome.
  • Generic advice. "Just use Docker" or "have you tried Kubernetes" without specifics. This sub is about the specific decisions, not the category.
  • Gatekeeping. Telling someone their factory isn't a "real" factory. Spaghetti factories are factories.

// THE PHILOSOPHY

Taiichi Ohno didn't keep the Toyota Production System secret. He wrote books about it. He invited competitors to tour the factory floor. His insight: sharing how you build doesn't eliminate your advantage — it compounds everyone's capability, including your own.

That's the model here. Share your production alpha openly. Link to your blueprint. Post your post-mortems. The factory must grow — and it grows fastest when every builder on the line can see how every other builder solved the same problem.

This subreddit runs itself. The checks are open-source. The moderation is automated. The rules are version-controlled. If you want to change how the factory works, open a PR.
